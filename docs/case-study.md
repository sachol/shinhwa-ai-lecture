# 사례 연구: 하네스 구조로 강의 자료 만들기

> 공인중개사 1인이 5명의 AI 에이전트 팀을 구성해 180분 특강 자료를 약 4시간 만에 만든 과정과 교훈.
>
> 이 문서는 다음 청중을 위해 작성됐다:
> - 강의·교육·콘텐츠 제작을 자주 하는 전문가
> - AI 자동화·에이전트 시스템에 관심 있는 실무자
> - "ChatGPT로 PPT 만들기"를 넘어, AI를 **시스템**으로 활용하고 싶은 사람

**TL;DR**
사람 1명 + Claude Code + 하네스(에이전트 정의 + 스킬) 구성으로, 사람 단독으로 4~6주 걸릴 강의 자료(리서치·커리큘럼·슬라이드·HTML·QA)를 **약 4시간**에 완성했다. 비결은 ChatGPT 단독 사용이 아니라 **역할 분담된 5명 AI가 협업**하는 구조였다.

---

## 1. 문제 — 왜 이렇게 만들어야 했나

### 1.1 강의 자료 제작의 전통적 병목

공인중개사 특강 자료 1세트(180분) 만들기 = 사람 1명이 풀로 매달려야 4~6주:

| 단계 | 사람 작업 | 평균 시간 |
|------|----------|----------|
| 시장·도구·법령 리서치 | 책·기사·법령 사이트 뒤지기 | 1주 |
| 커리큘럼 설계 | 청중 분석, 모듈 구성, 시간 배분 | 3일 |
| 슬라이드 본문 작성 | 모듈별 슬라이드 텍스트 | 1주 |
| 강사 스크립트 작성 | 슬라이드별 실제 말할 대본 | 4일 |
| HTML/PPT 빌드 | 디자인, 복사 버튼, 인쇄 CSS | 3일 |
| 사실 검증 / 일관성 검토 | 법령 조문·시간 배분·도구 정보 재확인 | 2일 |
| **합계** | | **약 28일** |

### 1.2 ChatGPT 단독 사용의 한계

"그냥 ChatGPT한테 시키면 되는 거 아냐?" — 직접 해봤다. 결과:

- **컨텍스트 잃어버림** — 모듈 5를 쓰는 동안 모듈 2에서 무슨 매물 예시를 썼는지 잊어버림
- **사실 검증 안 됨** — 법령 조문을 자신 있게 만들어냄 (환각). 검증은 사람이 따로 해야
- **일관성 깨짐** — 모듈마다 톤·용어·예시가 다름
- **자기 검토 못함** — "이 슬라이드가 학습 목표와 맞는가" 같은 메타 평가가 어려움

ChatGPT 단독으로는 **전 과정 6주의 시간을 3주로 단축** 정도. 부족하다.

### 1.3 해법 — AI를 한 명이 아니라 한 팀으로

**핵심 통찰:** 하나의 ChatGPT 인스턴스가 모든 일을 하려면 컨텍스트가 무너진다.
그 대신, **각자 역할이 다른 5명의 전문 AI**가 자기 영역만 깊게 파고, 결과를 파일로 주고받으면 어떨까?

이것이 **하네스(Harness)** 라는 개념이다. 도메인별로 전문 에이전트와 그들이 사용할 스킬을 미리 정의해두고, 새로운 작업이 들어오면 그 팀이 자동으로 출동한다.

## 2. 접근 — 하네스 구조 설계

### 2.1 하네스란 무엇인가

> **Harness = 전문 AI 에이전트들의 팀 + 그들이 사용할 스킬(가이드 문서) + 작업 조율 흐름**

비유: 잘 굴러가는 회사에는 마케팅·개발·디자인·QA가 따로 있고, 각 부서마다 "우리는 이렇게 일한다"는 매뉴얼이 있다. 사장은 새 프로젝트가 들어오면 적절한 팀을 꾸리고 매뉴얼을 따르게 한다.

하네스는 그것을 AI로 옮긴 구조다:
- **에이전트** = 각 분야 전문가 (마케팅·개발·디자인·QA에 해당)
- **스킬** = 각 분야의 매뉴얼 (자세히, 단계별로)
- **오케스트레이터** = 사장 (전체 흐름 조율)

### 2.2 이번 프로젝트의 팀 구성

청중 4개 주제 영역(AI 기본 / 매물 광고 / 고객 응대 / 시장·법령)을 고려해 다음 5명을 정의:

| 에이전트 | 역할 | 자기 매뉴얼(스킬) |
|----------|------|----------|
| **research-analyst** | 4개 주제 영역 리서치 (도구·사례·법령) | `realtor-curriculum-research` |
| **curriculum-designer** | 모듈 구성·시간 배분·학습 흐름 | `realtor-curriculum-design` |
| **content-writer** | 슬라이드 본문·강사 스크립트·실습 | `realtor-lecture-content` |
| **html-builder** | HTML 페이지 빌드 (한국어·인쇄·복사 버튼) | `realtor-html-lecture-build` |
| **qa-reviewer** | 경계면 교차 검증 | `realtor-lecture-qa` |

그리고 이 5명을 조율하는 **오케스트레이터 스킬** 1개 (`realtor-ai-lecture-orchestrator`).

### 2.3 협업 흐름 (Phase 1~6)

```
사용자 요청: "공인중개사 AI 특강 자료 만들어줘"
                ↓
        Phase 0: 컨텍스트 확인 (신규/재실행/부분 수정 판별)
                ↓
        Phase 1: 리서치 (research-analyst 4명 병렬)
                → 01_research_ai-basics.md
                → 01_research_property-analysis.md
                → 01_research_customer-service.md
                → 01_research_market-law.md
                ↓
        Phase 2: 커리큘럼 설계 (curriculum-designer 1명)
                → 02_curriculum.md
                ↓
        Phase 3: 콘텐츠 작성 (content-writer 3명, 모듈 묶음 병렬)
                → 03_content_module-1.md ~ module-6.md
                ↓
        Phase 4: HTML 빌드 (html-builder 1명)
                → output/index.html, output/practice.html
                ↓
        Phase 5: QA 검증 (qa-reviewer 1명, 7개 경계면 교차 비교)
                → 04_qa_report.md
                ↓
        Phase 6: 사용자 보고 + 피드백 수집
                ↓
        (피드백 발생 시) 부분 재실행 사이클
```

### 2.4 핵심 설계 결정 3가지

**① 병렬 vs 순차** — 리서치는 4개 영역이 서로 의존 없음 → **병렬**. 커리큘럼은 리서치 4건이 모두 끝나야 시작 가능 → **순차**. 콘텐츠 작성은 모듈 간 톤 일관성 위해 3명만 묶음 병렬(전부 병렬은 일관성 깨짐).

**② 파일 기반 데이터 전달** — 에이전트끼리 메모리·메시지가 아닌 `_workspace/` 디렉토리의 .md 파일로 주고받음. 이유: 중간 산출물 사후 검증·재실행에 필수. 사람이 파일을 보고 어디가 문제인지 진단 가능.

**③ QA는 "존재 확인"이 아닌 "경계면 교차 비교"** — 단순히 "모든 슬라이드가 있다"가 아니라 "리서치 노트의 도구가 콘텐츠 슬라이드에 정확히 등장하는가" 같은 7개 경계면 검증. 7개 경계면 중 1건이라도 어긋나면 Critical/Major.

## 3. 실행 — Phase 1~6 실제 흐름

### 3.1 Phase 1 (리서치, 약 50분)

- **research-analyst 4명을 한 메시지로 동시 호출** — 각자 자기 주제만 깊게
- 도구·사례 수집을 위해 WebSearch / WebFetch 각자 13~20번 호출
- 출처 URL + 확인일을 모든 인용에 첨부

**발견된 결정적 정보:**
- **AI 기본법 제31조 (2026.1.22 시행)** — AI 생성 콘텐츠 라벨 의무, 위반 시 3,000만원 과태료. 강의의 핵심 차별화 포인트가 됨
- **위반건축물 표기 의무 (2025.1.1)** — 광고문 자동 생성 프롬프트에 자동 삽입해야 할 필수 요소
- **개인정보보호법 제28조의8 국외이전 (2024.3.15 시행)** — ChatGPT가 미국 서버이므로 적용 대상

이 3개는 사람이 직접 리서치했다면 못 찾았을 가능성도 컸다. 4명이 병렬로 깊게 파니까 신호가 잡혔다.

### 3.2 Phase 2 (커리큘럼, 약 20분)

- 4개 리서치 노트를 모두 읽고, 입문~중급 듀얼 트랙 6모듈 설계
- 3:7 법칙 (개념 30% : 실습 70%) 시간 배분
- 모듈 간 의존성 다이어그램 + 위험·완화 표

### 3.3 Phase 3 (콘텐츠, 약 1시간)

- 모듈 6개를 1·2 / 3·4 / 5·6 묶음으로 나눠 content-writer 3명 병렬 호출
- **표준 가상 매물·고객을 사전 명시** — "마포구 공덕동 OO오피스텔, 고객A"를 모든 에이전트에게 미리 약속해서 모듈 간 일관성 확보
- 모듈별 슬라이드 본문 + 강사 스크립트 + 실습 + 금기 박스 + 정리 슬라이드 작성

### 3.4 Phase 4 (HTML 빌드, 약 40분)

- html-builder 1명이 모듈 6개 .md를 모두 읽고 단일 `index.html` + `practice.html` 빌드
- 한국어 타이포 (Pretendard) + 인쇄 CSS + 모바일 CSS + 복사 버튼 JS + 강사 모드 토글
- 결과: 초기 빌드 2,711줄·슬라이드 110장 → 이후 피드백 보완으로 **2,828줄·슬라이드 111장**, 복사 버튼 30개

### 3.5 Phase 5 (QA, 약 30분)

- qa-reviewer가 7개 경계면 검증 + WebSearch로 법령 4건 1차 출처 재확인
- **발견된 Critical 1건:** 모듈 1 Slide 1-6에서 "제17조 국외이전"이라 인용했으나 현행법은 **제28조의8**. 모듈 3에서는 정확히 인용했으므로 강의 내 자기모순. → 3개 파일 1줄씩 수정으로 해결
- **CONDITIONAL_PASS → PASS** 전환

### 3.6 Phase 6 (보고 + 피드백 사이클)

피드백 받아 다음 작업을 후속 진행:

| 피드백 | 처리 |
|--------|------|
| "강사 스크립트가 딱딱하다" | content-writer 3명 병렬 호출로 109개 블록 친근·구어체 일괄 변환 |
| "강사 모드에서 시간 압축 가이드가 필요" | content-writer 1명 호출로 모듈 4·5·6에 강사 진행 가이드 카드 추가 |
| "음성 안 될 때 백업이 없다" | 새 슬라이드 5-7B 추가 (음성 모드 실패 시 텍스트 대체 흐름) |
| "스크린샷 자료를 어떻게 채우지?" | screenshot-guide.md 작성 (29개 placeholder + 우선순위) |
| "이걸 GitHub에 올리고 메타 콘텐츠로 정리하고 싶다" | 본 문서 + README + 운영 문서 + Vercel 배포 |

각 피드백을 받을 때마다 **전체 자료를 다시 만들지 않고** 영향받은 에이전트만 부분 재실행. 이게 하네스의 진짜 장점.

## 4. 결과

### 4.1 정량

| 지표 | 값 |
|------|-----|
| 총 작업 시간 (사람 키보드 앞 시간) | **약 4시간** |
| 자동화된 에이전트 작업 (병렬 포함) | **약 3시간** |
| 생성된 슬라이드 | **111장** |
| 강사 스크립트 블록 | **106개** (HTML 동기화 기준) |
| 실습 시나리오 | **6개** |
| 법령 인용 (1차 출처 검증) | **17건+** |
| HTML 라인 수 | **2,828줄** (index.html) + 910줄 (practice.html) |
| 인용 출처 | **50+** (URL + 확인일 명시) |
| QA 검증 경계면 | **7개** |

### 4.2 정성

- **법령 안전성** — 17건 법령 인용 모두 1차 출처(법제처·국토부) 재검증 완료
- **청중 적합성** — 입문 60% / 중급 40% 균형, 모든 모듈에 듀얼 트랙 라벨
- **즉시 효용** — 모든 모듈에 "오늘 사무실 가서 쓸 1가지" 명시
- **운영성** — 강사 모드 토글로 한 자료를 강사용·청중용 동시 활용
- **재현성** — 동일 하네스로 다음 강의(예: 변호사·세무사 대상 AI) 자료도 같은 흐름으로 제작 가능

## 5. 교훈

### 5.1 ChatGPT 단독 vs 하네스 — 진짜 차이

| 차이점 | ChatGPT 단독 | 하네스 (5 에이전트) |
|--------|-------------|---------------------|
| 작업 깊이 | 표면적 ("일반론") | 깊음 (각 에이전트가 자기 영역만 파니까) |
| 사실 검증 | 사람이 따로 검증 | qa-reviewer가 자동 1차 출처 재검증 |
| 일관성 | 모듈마다 다름 | 표준 가상 매물·법령 인용 통일 |
| 재실행 | 처음부터 다시 | 영향받은 에이전트만 부분 재실행 |
| 비용 | 낮음 (월 $20) | 중간 (Claude Code Pro + 토큰) |
| 학습 곡선 | 즉시 사용 | 하네스 구성 1회 (이후 무한 재사용) |

**핵심:** 하네스는 "1회용 자료 1개"가 아니라 **"같은 도메인의 모든 자료를 만들 수 있는 공장"**을 만드는 일이다.

### 5.2 잘된 결정

- **표준 가상 매물 사전 약속** — 모듈 일관성 99% 달성 핵심
- **파일 기반 데이터 전달** — 사후 진단·부분 재실행 가능
- **QA 경계면 교차 비교** — Critical 1건(법령 자기모순) 즉시 발견
- **사용자 피드백 사이클** — 톤 변환, 가이드 카드 추가, 백업 슬라이드 추가 모두 부분 재실행

### 5.3 다시 한다면 바꿀 것

- **인쇄 CSS 검증** — 페이지당 슬라이드 14~24개 압축 가능성. 다음엔 인쇄 PDF로 한 번 더 검증
- **이미지 placeholder를 처음부터 정확한 텍스트로** — 사후 가이드 작성보다 처음 빌드 때 권장 캡처 텍스트 함께 작성
- **모듈 5·6 시간 배분 실측** — 풀버전 분량이 커리큘럼 명시 시간보다 김. 다음엔 슬라이드 수 × 평균 발화 시간을 처음부터 계산

### 5.4 도메인 확장 가능성

이 하네스의 5 에이전트 구조는 **다른 도메인 강의 자료**에도 거의 그대로 재사용 가능:

| 도메인 | 변경 필요한 부분 |
|--------|----------------|
| 변호사 대상 AI 강의 | 리서치 주제·법령(변호사법) / 표준 가상 사건 / 톤(변호사 적합) |
| 의사 대상 AI 강의 | 리서치 주제·법령(의료법·개보법 의료) / 표준 가상 환자 / 톤(의료진 적합) |
| 일반 직장인 AI 강의 | 리서치 주제 / 표준 시나리오 / 톤(가벼움) |

## 부록: 자주 묻는 질문

### Q. 코딩 모르는데 이걸 할 수 있나요?
A. 본 사례 작성자(공인중개사 신화)는 정통 개발자가 아니다. Claude Code 채팅창에 한국어로 "하네스 구성해줘", "특강 자료 만들어줘", "스크립트 톤 친근하게 바꿔줘"라고 말한 게 전부.

### Q. ChatGPT만으로는 안 되나요?
A. 가능하지만, 일관성·사실 검증·재실행이 어렵다. 일회용 자료 1개라면 ChatGPT로 충분. 같은 도메인 자료를 여러 번 만들 계획이라면 하네스가 시간 대비 압도적으로 유리.

### Q. 비용은 얼마나 드나요?
A. Claude Code Pro 월 구독 + 풀 사이클 1회당 약 $5~10 토큰. 사람 1주 인건비 대비 1/100~1/200 수준.

### Q. 법령·도구 정보가 시간 지나면 틀려질텐데?
A. **변경 이력 추적 + 업데이트 트리거** ([change-log.md](change-log.md))로 관리. AI 도구·법령이 변경되면 영향받은 모듈만 재실행. 풀 사이클 다시 안 돌려도 됨.
